Before you start · 2 min

Set up Claude Code

Claude Code is Anthropic’s AI coding tool that lives in your terminal. You talk to it in plain English; it reads your code, makes changes (with your approval), runs commands, and explains what it did. No prior AI-tools experience needed.

1 · Install (pick one)

# macOS / Linux / WSL (official installer)
curl -fsSL https://claude.ai/install.sh | bash

# or with npm
npm install -g @anthropic-ai/claude-code

# or with Homebrew (macOS)
brew install --cask claude-code

2 · Start a session in any project

cd your-project
claude          # log in on first run, then type /help
Our starter for today: create a file called game.py with the scaffold in the “Starter project” section below, then run claude in that folder. You’ll turn it into a working game in the 7 steps.
The walkthrough

7 steps, 15 minutes

Do these in order. Tick each one off as you go — your progress is saved on this device.

0 / 7 done
15:00
1

Set up & orient

2 min · setup

Open your terminal, cd into the folder with game.py, and run claude. Log in when prompted. Type /help to see what’s available, then press Esc to return to the prompt.

You’re learning: Claude Code is a conversation in your project folder — it can see the files there.

2

Talk to your codebase

2 min · prompting

Before changing anything, ask Claude to explain the file. Use @ to pull a file into the conversation:

Paste this prompt

@game.py What does this file do? Walk me through it step by step, and flag anything that looks incomplete. Assume I'm a beginner who has never seen this code.

You’re learning: let the tool orient you first. @file gives Claude exact context instead of guessing.

3

Make your first change

3 min · iteration

Give one clear, specific instruction. Notice the pattern: what I have + what’s missing + a constraint.

Paste this prompt

I have a Python guessing game in @game.py. Right now play() asks for a guess only once and exits. Add a while loop so it keeps asking until the user guesses the secret number. Keep it under 30 lines and add one short comment per section.

Claude will ask permission before it edits the file — read the diff and approve it. Then run it:

python game.py

You’re learning: Claude proposes, you approve. Always read the change before saying yes, then run it to see it work.

4

Iterate

2 min · iteration

Small follow-ups are the heart of the workflow. Add a feature:

Now tell the player whether each guess is too high or too low. Don't change anything else.

Hit a crash? Paste the exact error — Claude can’t see your screen:

It crashed with: ValueError: invalid literal for int() with base 10: ''. That happens when I press Enter without typing anything. Fix just that bug (catch it and ask again) without rewriting the rest of the game.

You’re learning: “don’t change anything else” keeps Claude from over-engineering. Paste errors verbatim, never paraphrased.

5

Prompt better + Plan Mode

3 min · planning

For anything bigger than a one-liner, have Claude plan first. Plan Mode lets it research and propose without touching files. Turn it on three ways: press Shift+Tab to cycle to plan, type /plan, or start with claude --permission-mode plan.

Paste this prompt (after entering plan mode, or just start it with /plan)

Before writing any code, outline a step-by-step plan to add a high-score feature: save the lowest number of guesses to a file called scores.txt and show it at the start of each game. List the plan as numbered steps, note any risks, and wait for my approval before changing anything.

For a genuinely hard problem, add the keyword ultrathink to your prompt for deeper reasoning on that turn.

You’re learning: plan before you build. A weak prompt (“make it better”) gets weak results; specifics and a plan get great ones.

6

Review your work + make a skill

2 min · skills

Claude Code ships with skills — reusable commands you trigger with a slash. Two come built in for reviewing your work:

/code-review        # reviews your current changes for bugs + cleanups
/security-review    # checks your changes for security issues

You can also just ask, in plain English:

Review the change you just made. List any bugs, edge cases that could crash it, and lines that are more complicated than they need to be. Then tell me which are worth fixing now.

Make your own skill: create .claude/skills/game-review/SKILL.md (or a one-file .claude/commands/game-review.md) and you’ll have a /game-review command. Let Claude build it for you:

Create a skill called game-review for this project. It should read game.py and check for common beginner bugs (missing input validation, bare except clauses, magic numbers), then print a short bulleted report. Save it so I can run it again with /game-review.

You’re learning: “skills” like /plan and /code-review are real, reusable commands — and you can write your own.

7

Make it yours & wrap up

1 min · memory

Run /init and Claude writes a CLAUDE.md — persistent project notes it reads at the start of every session. Use /clear to start a fresh conversation between unrelated tasks. That’s the whole loop. Bring what you built to Friday’s demo (2–3 min) and share one thing the AI got wrong.

You’re learning: CLAUDE.md is the tool’s memory of your project. Good context in = good help out.

The starter project

Guess the Number

A tiny CLI game — zero setup, runs with one command, short enough to read every line. Create game.py with this scaffold (it’s intentionally incomplete; you’ll finish it in the steps):

# game.py — starter scaffold for the Week 1 tutorial
import random

def play():
    secret = random.randint(1, 10)
    # TODO: ask the user to guess, check it, and repeat until correct

play()

The natural build arc: complete the loop (Step 3) → add too-high / too-low hints and fix the empty-input crash (Step 4) → plan and add a high-score file (Step 5) → review it (Step 6). Infinite room to keep going.

Keep these handy

Prompt library

Six patterns worth saving. Good prompts are specific and give context.

Understand code

What does @path/to/file do? Walk me through it step by step and flag anything incomplete or buggy. Assume I'm new to this code.

Specific change ([what I have] + [what's missing] + [constraint])

In @file, the X function does A but not B. Add B by doing C. Keep it small and add one comment per section. Don't touch anything else.

Fix a crash

It crashed with: [paste the full error]. This happens when [what you did]. Fix just that bug without rewriting the rest.

Plan before building

Before writing any code, outline a numbered plan for [feature]. Note risks and edge cases, then wait for my approval before changing anything.

Self-review

Review the change you just made for bugs, edge cases, and anything more complicated than it needs to be. Tell me which findings are worth fixing now.

Make a reusable skill

Create a skill called [name] for this project that [does X]. Save it so I can run it again with /[name].
The rules that matter most

Three golden rules

01

Read every line

AI is confident even when it’s wrong. If you can’t explain a block out loud, ask Claude to explain it before you ship it. Read the diff, not just the new code.

02

Test it yourself

“Should work” isn’t done. Run it. Try bad input — empty string, negative, a letter. The happy path is the easy 80%.

03

Commit small & often

Commit before and after AI changes (git commit -m "before: X"). It’s your instant undo button if a change goes sideways.
